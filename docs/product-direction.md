# 产品方向

## 一句话概述

Touckey 将 Android 手机变成面向 Windows 和 macOS 的 Bluetooth HID 键盘与触摸板，桌面端无需安装任何伴生软件。
长期来看，它会逐步演进为一个受 QMK 启发的可编程输入面板，但这不是首发版本的前提。

## 产品定位

Touckey 不应只是一个“远程鼠标应用”。
它应该先成为一个可靠、顺手、零配置的手机控制面板，再逐步演进成可编程输入面板：

- 足够偏键盘，能承载快捷键与常用控制动作
- 足够偏触控，让体验更像真正的触摸板，而不是一个巨大的鼠标按钮
- 足够简单，让桌面端无需驱动、无需辅助程序即可使用

## 核心产品支柱

- 将触控与键盘整合成一个产品，而不是两个割裂的模式
- 基于标准 Bluetooth HID，实现桌面端零配置
- 优先服务高频快捷操作，而不只是普通文本输入
- 采用手机原生交互设计，而不是伪装成硬件键盘配置器
- 在基础体验成熟后，再把受 QMK 启发的行为模型适配到手机交互中

## 产品阶段

### 阶段 A：首发产品

目标是先做出一个可靠的手机输入遥控器，而不是配置系统：

- 键盘输入
- 相对指针移动
- 点击、右击、拖拽、滚动
- 媒体与系统常用快捷动作
- 少量预设快捷按钮或预设手势

这一阶段不要求用户自己搭层、写宏或配置复杂行为。

### 阶段 B：轻量可配置

在基础体验稳定后，再开放有限配置能力：

- 调整快捷按钮布局
- 选择或切换预设动作集
- 自定义常用快捷键和手势映射
- 保存本地个人偏好

这一阶段仍然以“好用的控制器”为核心，而不是“完整键盘编程器”。

### 阶段 C：进阶可编程

当基础输入、连接稳定性和跨平台行为都足够成熟后，再引入受 QMK 启发的能力：

- 层
- 宏
- tap-hold
- one-shot 修饰键
- combos 与 chords

QMK 气质应该是 Touckey 的长期方向，而不是首版负担。

## 目标用户

- 键盘和自动化爱好者
- 需要沙发场景或便携控制面的桌面/笔记本用户
- 依赖快捷键、窗口管理和应用切换的高频用户
- 希望先获得高效控制体验，并在后续获得可编程能力的人

## 应该重点打磨的体验

- 低摩擦地触发快捷键和常用动作
- 通过触控区域完成指针移动、滚动、拖拽和常用手势
- 将手势映射到应用切换、任务视图、Mission Control 或媒体控制等系统动作
- 在同一会话中自然组合软键区与触控手势
- 在需要时逐步开放更强的自定义能力，而不是一开始就暴露复杂配置

## 产品定义

Touckey 应将自己暴露为一个标准 Bluetooth HID 组合设备，并对外提供：

- 键盘输入
- 相对鼠标移动
- 滚轮与横向滚动
- 音量、媒体等 consumer control 按键
- 少量可直接使用的预设快捷动作

首发版本优先做“稳定输入 + 好用预设”，而不是完整可编程系统。

在这些标准 report 之上，产品中长期再逐步提供更高层的可编程模型：

- 层
- tap-hold 行为
- combos 与 chords
- 宏
- one-shot 修饰键
- 手势到动作的映射

## 早期版本的非目标

- 桌面端伴生软件
- 自定义桌面驱动
- 完整远程桌面或屏幕串流
- 首发即提供完整 QMK 风格可编程系统
- 伪装成原生 MacBook 触摸板或 Windows Precision Touchpad
- 在经典 HID 组合输入尚未稳定前，过早投入 BLE 专用协议工作
- 游戏手柄支持

## 为什么不优先追求原生触摸板协议

第一阶段最重要的是行为可靠，而不是协议“纯正”。
如果预设动作、点击、滚动、拖拽和常用快捷操作已经足够顺手，它就已经具备价值。
过早追求平台原生级的精密触摸板仿真，只会显著提高复杂度与兼容性风险。

## 工作原则

先把它做成一块可靠的蓝牙控制面板，再把它演进成一块具有 QMK 气质的输入控制台。
