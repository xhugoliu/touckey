# 架构草图

## 目标

在正式写实现代码之前，先定义软件整体形态。
本文档刻意保持在高层，不进入具体实现细节。

## 系统边界

应用完全运行在 Android 端，并将自己暴露为一个标准 Bluetooth HID 外设。
桌面端只应看到标准输入设备，不应依赖额外软件。

## 建议的 Report 类型

第一版建议从三类 report 开始：

- keyboard report
- relative mouse report
- consumer control report

这样既能保持较好的兼容性，也足以支撑较丰富的交互模型。

## 建议的子系统

### 1. 应用外壳层

职责：

- 应用启动
- 权限管理
- 导航
- 前台服务生命周期接入

### 2. HID 服务层

职责：

- 托管 `BluetoothHidDevice` 生命周期
- 注册和注销 HID app
- 对外暴露连接状态
- 通过前台服务支撑应用在普通后台场景下继续工作

### 3. 会话控制器

职责：

- 跟踪当前 host 设备
- 管理连接与断开流程
- 串行化发送 report
- 将应用层动作转换成 HID report

### 4. 键位映射引擎

职责：

- 在首发阶段承载预设快捷动作和基础映射
- 在后续阶段逐步演进为更完整的输入行为引擎
- 层
- 宏
- tap-hold 规则
- one-shot 修饰键
- combos 与 chords

输出：

- 如“发送按键”“保持修饰键”“执行宏”“切换层”之类的抽象动作

说明：

- 第一版可以只实现“按钮或手势 -> 动作”的轻量映射
- 不必在首发版本一次做完整 QMK 式行为系统

### 5. 手势引擎

职责：

- 识别指针与多点触控手势
- 将手势规范化为应用层动作
- 保持手势逻辑与蓝牙传输层解耦

输出：

- 指针移动
- 点击
- 拖拽开始与结束
- 滚动
- 横向滚动
- 触发宏或快捷动作

### 6. 动作分发器

职责：

- 接收键位映射引擎与手势引擎产生的动作
- 将其映射到键盘、鼠标或 consumer control report
- 统一处理顺序、延时与释放行为

### 7. 配置存储

职责：

- 持久化层与手势映射
- 支持未来的导入与导出
- 将默认配置与用户修改隔离开

### 8. 编辑器界面

职责：

- 用适合手机的方式呈现层与宏
- 在不过度增加负担的前提下暴露手势映射能力
- 让高级功能可发现，但不变成使用门槛

## 建议的事件流

1. 用户触发软键区或触控区。
2. UI 产出输入事件。
3. 键位映射引擎或手势引擎解释该事件。
4. 动作分发器将结果转换为一个或多个 HID report。
5. HID 服务将 report 发送给当前连接的 host。

## 现实约束

- Android 端的 HID 外设行为对生命周期与前台状态较为敏感
- Windows 与 macOS 对滚轮、横向滚动和部分快捷键组合的解释可能不同
- 手势能力应建立在标准 HID 行为之上，而不是建立在对原生触摸板协议的假设上

## 技术决策原则

优先：

- 标准 HID 兼容性
- 显式动作映射
- 手势识别与 HID 传输之间清晰的内部边界

避免：

- 将 UI 手势直接耦合到原始蓝牙调用
- 把复杂行为塞进单一 Activity
- 过早投入那些无法提升首版可用性的协议实验
